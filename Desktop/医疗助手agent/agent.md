# OpenEvidence 自进化医疗循证Agent - 项目说明文档

## 一、项目概述

OpenEvidence 是一个面向临床医生的自进化医疗循证证据问答助手，覆盖高血压、高血脂、2型糖尿病、心脑血管合并症四大疾病领域。系统基于 RAG（检索增强生成）架构，引入 LangChain Agent 框架实现工具调用，并构建了三层自进化机制（知识自进化、策略自进化、推理自进化）和安全防护体系。

### 核心特性

- **Agent 工具调用**：LLM 自主决策调用 4 种工具（本地检索、外部搜索、证据评估、查询分类）
- **实时外部检索**：PubMed + EuropePMC 双源 API，当本地证据不足时自动补充最新文献
- **三层自进化**：知识缺口检测与补充、策略自博弈优化、推理原则蒸馏与强化
- **安全防护**：4 级风险评级、审计日志、统计显著性检验、人工审核（Human-in-the-loop）
- **混合检索**：BM25 + 向量检索（BGE-large-zh-v1.5, 1024维）加权融合 + CrossEncoder 重排
- **GPU 加速**：RTX 4060 Laptop GPU，嵌入生成与重排序均使用 CUDA

## 三、Agent 系统设计

### 3.1 Agent Loop 工作流程

Agent 采用 LangChain 的 tool calling 机制，核心循环如下：

```
用户提问
    │
    ▼
┌──────────────┐
│ 领域检查      │ ──→ 不在医疗领域 → 拒绝回答
└──────┬───────┘
       │
       ▼
┌──────────────┐     ┌──────────────────────────┐
│ Agent Loop   │────→│ LLM 决策调用哪个工具      │
│ (最多5轮)    │     │ (query_classifier优先)    │
└──────┬───────┘     └──────────────────────────┘
       │
       ▼
┌──────────────────────────────────────────────────┐
│  轮次1: query_classifier → 分类+疾病识别          │
│  轮次2: local_search → 本地知识库检索             │
│         (相似度<0.7? → 记录知识缺口)              │
│  轮次3: external_search → PubMed+EuropePMC补充   │
│         (自动保存到数据库供未来本地检索)           │
│  轮次4: evidence_evaluator → 评估证据充分性       │
│  轮次5: 证据充分 → 退出循环                       │
└──────────────────────────────────────────────────┘
       │
       ▼
┌──────────────────────────────────────────────────┐
│  构建最终Prompt:                                  │
│  1. 检索到的证据片段                              │
│  2. 从推理原则库检索匹配原则注入                   │
│  3. 系统提示词 (回答结构要求)                      │
└──────────────────────────────────────────────────┘
       │
       ▼
┌──────────────────────────────────────────────────┐
│  SiliconFlow LLM 流式生成答案                     │
│  (SSE 推送: status → chunks → text → refs → done)│
└──────────────────────────────────────────────────┘
       │
       ▼
┌──────────────────────────────────────────────────┐
│  后台记录 (静默执行):                              │
│  1. 经验记忆库记录 (查询/策略/结果/评分)           │
│  2. 知识缺口记录 (如检测到)                       │
│  3. 安全审计日志                                  │
│  4. 推理原则置信度更新 (如使用了原则)              │
└──────────────────────────────────────────────────┘
```

### 3.2 四个 Agent 工具

| 工具名 | 功能 | 输入 | 输出 |
|--------|------|------|------|
| `query_classifier` | 分类查询类型和识别疾病 | 用户问题 | 类型(6种)+疾病标签 |
| `local_search` | 搜索本地知识库 | 查询关键词 | 证据片段列表+相似度+缺口检测 |
| `external_search` | 实时搜索PubMed+EuropePMC | 查询关键词 | 外部文献列表(自动入库) |
| `evidence_evaluator` | 评估证据充分性 | 证据摘要+查询 | 充分性+评分+证据等级 |

**查询类型分类**（6种）：治疗推荐(treatment)、诊断标准(diagnosis)、药物相互作用(drug_interaction)、预后评估(prognosis)、指南解读(guideline_interpretation)、机制原理(mechanism)

### 3.3 SSE 流式通信

Agent 模式使用 7 种 SSE 消息类型实现实时通信：

| 消息类型 | 说明 | 前端展示 |
|----------|------|----------|
| `status` | 状态更新 | 状态栏+旋转图标 |
| `tool_call` | Agent调用工具 | 工具调用面板 |
| `tool_result` | 工具执行结果 | 工具调用面板 |
| `chunks` | 检索到的证据 | 证据卡片(本地/外部标签) |
| `text` | 答案文本片段 | 答案区域逐字显示 |
| `references` | 参考文献附录 | 文末引用列表 |
| `done` | 回答完成 | 结束状态 |

## 四、三层自进化机制

### 4.1 第一层：知识自进化

**目标**：自动检测本地知识库的覆盖盲区，并实时补充外部文献。

**机制**：
1. 每次 `local_search` 返回结果时，检查最高相似度
2. 若相似度 < 0.7，判定为"知识缺口"，记录到 `table_knowledge_gap` 表
3. Agent 自动调用 `external_search` 从 PubMed + EuropePMC 获取文献
4. 获取到的文献自动保存到数据库，未来可直接本地检索
5. `GapLogger` 分析高频缺口领域，生成批量获取建议

**关键文件**：`backend/agent/gap_logger.py`

**数据表**：`table_knowledge_gap`

| 字段 | 说明 |
|------|------|
| gap_id | 缺口ID |
| query | 触发缺口的查询 |
| gap_type | 缺口类型(no_result/low_similarity/outdated) |
| max_similarity | 本地检索最高相似度 |
| domain | 疾病领域 |
| external_source | 外部补充来源(pubmed/europepmc/both) |
| external_results_count | 外部检索到的文献数 |
| resolved | 是否已解决 |
| timestamp | 时间戳 |

### 4.2 第二层：策略自进化

**目标**：通过自博弈评估自动优化检索策略参数。

**机制**：
1. **经验记录**：每次问答后自动记录到 `table_agent_experience` 表
2. **自博弈评估**：
   - 从数据库随机抽取文献，让 LLM 生成测试题
   - 用 3 种不同策略分别回答：
     - 默认策略：`bm25_weight=0.4, vector_weight=0.6, threshold=0.1, top_k=10, top_n=8`
     - 优化策略：历史最优参数
     - 激进策略：`bm25_weight=0.2, vector_weight=0.8, threshold=0.05, top_k=15, top_n=12`
   - 自动评分（证据命中率、引用数、相似度加权）
   - 选出最优策略，更新到 `table_strategy_optimization` 表
3. **策略推荐**：新查询时，根据查询类型从优化表检索最优策略
4. **跨查询迁移**：使用 Jaccard 相似度从历史经验中复用成功策略

**关键文件**：`backend/agent/memory_bank.py`, `backend/agent/strategy_evolution.py`

**自动评分规则**：
```
基础分 = 0.3
+ 指南证据 × 0.3
+ RCT证据 × 0.25
+ Meta分析 × 0.25
+ min(证据数/5, 1) × 0.2
- 相似度 < 0.5 × 0.3
最高分 = 1.0
```

### 4.3 第三层：推理自进化

**目标**：从高分经验中蒸馏可复用的抽象推理原则，指导后续问答。

**机制**（借鉴 EvolveR 框架，ICML 2026）：

1. **离线蒸馏** (`distill_principles`)：
   - 筛选评分 >= 0.7 的高分经验
   - 按查询类型分组，每组至少 3 条经验
   - LLM 分析共同模式，提取"当...时，应该..."格式的抽象原则
   - 每条原则初始置信度 = 0.5

2. **在线检索** (`retrieve_principles`)：
   - 新查询时，按查询类型 + 疾病标签匹配原则
   - 按置信度降序排列，注入到最终答案生成的 Prompt 中
   - 格式：`## 推理原则指导（从历史成功经验蒸馏）`

3. **策略强化** (`update_principle_confidence`)：
   - 每次使用原则后，根据答案评分更新置信度
   - 贝叶斯式更新：`new = 0.5 × (1-weight) + success_rate × weight`
   - `weight = min(usage_count / 10, 1.0)`（使用次数越多，数据驱动权重越大）
   - 成功额外加 0.05，失败乘以 0.95 衰减

**关键文件**：`backend/agent/reasoning_evolution.py`

**蒸馏 Prompt 示例**：
```
你是一个医疗AI系统的推理策略分析专家。
以下是treatment类型的5条高分问答经验（评分>=0.7）...
蒸馏出3-5条可复用的抽象推理原则。
每条原则用一句话描述，以"当...时，应该..."的格式
```

**已蒸馏出的原则示例**（当前5条）：
1. 当查询涉及治疗方案的临床决策时，应该采用BM25与向量检索的加权组合（0.4:0.6）平衡关键词与语义匹配
2. 当检索结果包含多篇1A级证据时，应该优先展示8篇高一致性证据以强化临床推荐力度
3. 当处理药物选择类查询时，应该设置0.1的重排阈值来过滤低相关性的片段
4. 当证据等级跨度超过两级时，应该在最终输出中保留但明确标注证据梯度
5. 当查询包含"首选"等确定性表述时，应该限制外部检索并依赖本地高质量指南证据

## 五、安全防护体系

### 5.1 四级风险评级

| 风险等级 | 说明 | 触发场景 | 处理方式 |
|----------|------|----------|----------|
| LOW | 低风险 | 经验记录、缺口记录 | 自动通过 |
| MEDIUM | 中风险 | 策略参数调整 | 记录审计，需人工审核 |
| HIGH | 高风险 | 推理原则蒸馏/变更 | 记录审计，需人工审核 |
| CRITICAL | 极高风险 | 知识库删除、证据等级变更 | 必须人工审核后才生效 |

### 5.2 安全阈值配置

| 阈值 | 默认值 | 说明 |
|------|--------|------|
| `min_samples_for_strategy_update` | 5 | 策略更新最少需要5次经验 |
| `min_samples_for_principle_distill` | 3 | 原则蒸馏最少需要3条高分经验 |
| `min_confidence_for_auto_apply` | 0.8 | 原则自动应用的最低置信度 |
| `max_strategy_change_per_update` | 0.3 | 单次策略参数最大变化幅度(30%) |
| `significance_threshold` | 0.1 | 策略改善的统计显著性阈值 |

### 5.3 统计显著性检验

使用 Cohen's d 效应量检验策略改善是否具有统计显著性，防止从噪声中学习：

```
Cohen's d = (new_mean - old_mean) / pooled_std

|d| < 0.2  → 不具备统计显著性，不更新策略
|d| >= 0.2 且 effect > 0 → 改善显著，更新策略
|d| >= 0.2 且 effect < 0 → 策略退化，不更新
```

### 5.4 审计日志

所有自进化操作均记录到 `table_evolution_audit` 表，包含：
- 操作类型（strategy_update / principle_distillation / experience_record 等）
- 变更前后值（JSON格式）
- 风险等级
- 审核状态（待审核 / 已审核 / 自动通过）
- 审核人 + 审核备注

**关键文件**：`backend/agent/safety_guard.py`

## 十、如何使用

### 10.1 启动服务

```bash
# 启动后端（需等待约30秒加载嵌入模型）
cd 医疗助手agent
E:\anaconda\envs\AI\python.exe -m uvicorn backend.main:app --host 0.0.0.0 --port 8000

# 启动前端
cd frontend
npm run dev
```

### 10.2 使用 Agent 问答

1. 打开 `http://localhost:5173`
2. 进入"循证问答"页面
3. 右上角切换到 **Agent 模式**（魔棒图标）
4. 输入医疗问题，例如："高血压合并糖尿病的患者首选什么降压药？"
5. 观察工具调用面板：Agent 会依次调用 query_classifier → local_search → external_search（如需要）→ evidence_evaluator
6. 答案生成后，系统自动在后台记录经验

### 10.3 查看自进化状态

1. 点击导航栏 **"自进化监控"**
2. **经验记忆库**：查看每次问答的经验记录、评分、工具调用详情
3. **策略自博弈**：点击"触发自博弈评估"，系统自动生成测试题、比较3种策略、更新最优参数
4. **推理原则**：点击"蒸馏推理原则"，系统从高分经验中提取可复用原则（需先积累足够高分经验）
5. **知识缺口**：查看本地知识库的覆盖盲区和批量获取建议
6. **安全审计**：查看所有自进化操作的审计日志，对待审核操作进行通过/拒绝

### 10.4 自进化触发条件

| 进化类型 | 触发方式 | 最低数据要求 |
|----------|----------|-------------|
| 知识缺口记录 | 自动（每次Agent问答相似度<0.7时） | 无 |
| 经验记录 | 自动（每次Agent问答后） | 无 |
| 策略自博弈 | 手动触发（监控面板按钮） | 数据库中有文献 |
| 推理原则蒸馏 | 手动触发（监控面板按钮） | 3条以上评分>=0.7的经验 |
| 原则置信度更新 | 自动（每次使用原则后） | 已有蒸馏出的原则 |
| 安全审计 | 自动（所有自进化操作） | 无 |

**Ground Truth来源**：中国高血压防治指南2024、中国2型糖尿病防治指南2024、中国血脂管理指南2024、ESC/EAS指南、ADA标准、EMPA-REG/CANVAS/DECLARE试验等。

| LDL-C目标值 | 0.636 | 完整性0.89，正确列出风险分层对应的目标值 |

### 12.4 失败原因分析

**原因1：LLM调用异常 (6/15题)**
- 6道题在170-240ms内返回，答案为"[LLM调用异常：]"
- 根因：SiliconFlow API在高频请求时出现速率限制或超时
- 影响：这些题目完全没有获得Agent的回答

**原因2：答案关键信息覆盖不足 (4/15题)**
- Agent成功生成了答案，但关键数值/药物名未完全覆盖
- 例如：高血压合并糖尿病题，答案正确提到ACEI/ARB，但未提到"130/80mmHg目标"

**原因3：证据检索不足 (2/15题)**
- 部分题目本地知识库覆盖不足，相似度低于0.7
- Agent调用了external_search但API返回结果有限

## 十三、MCP与Skill集成可行性分析

### 13.1 MCP (Model Context Protocol)

**核心价值**：标准化AI与外部工具/数据源的连接，将M个应用×N个工具的集成复杂度从O(M×N)降至O(M+N)。

**当前项目评估**：

| 维度 | 评估 |
|------|------|
| 开发复杂度 | **高**：需为4个工具各实现MCP Server + JSON-RPC传输层 |
| 收益 | **低**：当前是单一自包含应用，4个工具都是内部Python函数，无跨平台复用需求 |
| 风险 | **高**：无成熟医疗MCP Server参考；供应链安全风险；医疗合规风险 |
| 替代方案 | LangChain已提供`langchain-mcp-adapters`，未来有跨平台需求可低成本接入 |

**结论：当前不推荐引入MCP。** MCP的核心价值在于跨平台/跨应用复用和标准化连接外部系统（如HIS/PACS/LIS）。对于当前4工具的自包含RAG Agent，引入MCP会增加不必要的协议抽象层，增加调试复杂度，却无法带来实质收益。未来若需对接医院信息系统时再考虑。

### 13.2 Skill机制

**核心价值**：通过三层渐进式披露（元数据→指令→资源）优化Token效率，将操作SOP模块化管理。Anthropic称"MCP connects data, Skills teach what to do with that data"。

**当前项目评估**：

| 维度 | 评估 |
|------|------|
| 开发复杂度 | **中**：可实现轻量版Skill加载机制 |
| 收益 | **中高**：解决系统提示词膨胀；推理原则模块化管理；工作流程SOP化 |
| 风险 | **低**：纯文本驱动，无安全风险 |
| 关键限制 | 完整Skill系统需文件系统+代码执行环境，当前API模式不支持Script功能 |

**推荐做法**：借鉴Skill的渐进式披露思想，在现有架构内实现轻量改造：

1. **将推理原则改造为Skill格式**：现有"当...时，应该..."格式的原则，可重构为SKILL.md格式，实现元数据索引 + 按需加载
2. **将系统提示词拆分为多个Skill**：工作流程规则、证据等级说明、引用格式规范等，按需注入而非全量常驻
3. **保留LangChain Tool Calling**：工具调用层不需要改动

### 13.3 分阶段建议

| 阶段 | 建议 | 理由 |
|------|------|------|
| **当前** | 不引入MCP，采用Skill思想优化提示词 | 复杂度低，收益明确 |
| **短期优化** | 将推理原则和系统提示词模块化为Skill格式 | 解决提示词膨胀，提升Token效率 |
| **中期（如有需求）** | 通过langchain-mcp-adapters将external_search封装为MCP Server | 便于被Claude Desktop等工具复用 |
| **长期（对接医院系统时）** | 引入MCP对接HIS/PACS/LIS | MCP的标准化价值此时才能真正体现 |

**一句话结论**：对于当前4工具的医疗RAG Agent，**MCP暂无必要**（复杂度远大于收益），**Skill的思想值得借鉴**（渐进式披露优化提示词管理），但完整Skill系统受限于API模式无法充分发挥。优先级应是：优化现有工具质量（如将规则型query_classifier和evidence_evaluator升级为LLM驱动）> 引入Skill思想优化提示词 > 考虑MCP。
