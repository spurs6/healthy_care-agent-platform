# OpenEvidence 评测系统说明文档

## 一、评测概述

OpenEvidence 评测系统用于量化评估 RAG（检索增强生成）系统在临床循证医学问答中的表现。系统覆盖四大疾病领域（高血压、高血脂、2型糖尿病、心脑血管合并症），从**检索能力**和**生成质量**两个维度对系统进行全面评测。

### 评测流程

```
测试集准备 → RAG检索 → LLM生成答案 → 指标计算 → 结果持久化 → 历史追踪
     ↓           ↓          ↓           ↓          ↓
  test_set/   BM25+向量   SiliconFlow  4项指标   SQLite
   JSON文件   混合检索     API调用      自动计算   table_eval_history
```

## 二、测试集设计

### 2.1 两类测试集

系统支持两种互补的测试集来源，分别评估 RAG 系统的不同能力层面：

| 测试集类型 | 来源 | 评估目标 | 示例文件 |
|-----------|------|---------|---------|
| **域内测试集** | 基于知识库自动生成 | 检索系统是否正确找回知识库中的内容 | `auto_generated_*.json` |
| **域外测试集** | 基于权威指南手工构建 | 系统的通用医学知识能力和临床推理能力 | `clinical_evidence_test_v3.json` |

### 2.2 域内测试集（自动生成）

**生成原理**：从数据库 `table_article_meta` 表中随机抽取含摘要的文献，使用 LLM 基于文献摘要内容生成问答题对。答案完全基于文献内容，不编造信息。

**生成流程**：
1. 从数据库随机抽取 N×3 篇含摘要的文献
2. 对每篇文献，构造 prompt 让 LLM 生成一道临床问答题
3. LLM 返回 JSON 格式的 `{"query": "...", "ground_truth": "...", "source": "..."}`
4. 解析并验证 QA 对格式
5. 保存到 `test_set/auto_generated_{timestamp}.json`

**适用场景**：验证 RAG 系统的检索召回率和引用准确率。因为标准答案来源于知识库文献本身，如果检索系统工作正常，答案中的关键信息应当能被检索到并正确引用。

### 2.3 域外测试集（手工构建）

**构建原理**：基于中国临床指南和国际 RCT 试验证据，由人工编写 15 道临床问答题，覆盖四大疾病领域的核心临床决策点。

**测试集结构**：

```json
{
  "query": "临床问题",
  "ground_truth": "基于权威指南的标准答案",
  "source": "答案来源（指南名称+年份）",
  "expected_chunks": null
}
```

**题目分布**：

| 疾病领域 | 题数 | 考察重点 |
|---------|------|---------|
| 高血压 | 3 | 合并症用药选择、急症处理原则 |
| 糖尿病 | 4 | 一线用药、心血管获益证据、肾病分期、HbA1c标准 |
| 高血脂 | 2 | LDL-C目标值分层、他汀不良反应 |
| 心脑血管 | 6 | 抗凝策略、心梗诊断、二级预防、心衰治疗、DAPT时长、代谢综合征 |

**Ground Truth 来源**：

| 来源 | 引用频次 |
|------|---------|
| 中国高血压防治指南2024 | 3题 |
| 中国2型糖尿病防治指南2024 | 3题 |
| 中国血脂管理指南2024 | 2题 |
| 中国心力衰竭诊断和治疗指南2024 | 1题 |
| ESC/EAS指南 | 2题 |
| ADA糖尿病医学诊疗标准 | 2题 |
| EMPA-REG/CANVAS/DECLARE试验 | 1题 |
| ONTARGET试验 | 1题 |

### 2.4 测试集数据格式

所有测试集统一采用 JSON 数组格式，每条数据包含以下字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `query` | string | 是 | 临床问题 |
| `ground_truth` | string | 是 | 标准答案 |
| `source` | string | 否 | 答案来源标识 |
| `expected_chunks` | array | 否 | 预期检索到的文档ID列表（用于计算检索精准率） |

## 三、评测指标

### 3.1 指标体系总览

系统采用 4 项核心指标，分别评估 RAG 系统的不同环节：

```
┌─────────────────────────────────────────────────────────┐
│                    RAG 评测指标体系                       │
│                                                          │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │ 证据忠实度    │  │ 引用准确率    │  │ 检索精准率    │  │
│  │ Faithfulness │  │ Citation Acc │  │ Context Prec │  │
│  │              │  │              │  │              │  │
│  │ 答案是否忠实  │  │ 引用格式是否  │  │ 检索结果是否  │  │
│  │ 于检索到的    │  │ 正确指向真实  │  │ 与问题高度    │  │
│  │ 证据内容      │  │ 检索到的文档  │  │ 相关          │  │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘  │
│         │                 │                 │           │
│         └────────┬────────┘                 │           │
│                  │                          │           │
│           生成质量评估                  检索质量评估      │
│                                                  │      │
│                                    ┌─────────────┴────┐ │
│                                    │   答案相关性      │ │
│                                    │ Answer Relevance │ │
│                                    │                  │ │
│                                    │ 答案与标准答案的  │ │
│                                    │ 关键词重合度      │ │
│                                    └──────────────────┘ │
│                                           综合评估       │
└─────────────────────────────────────────────────────────┘
```

### 3.2 指标1：证据忠实度（Faithfulness）

**定义**：衡量 LLM 生成的答案是否忠实于检索到的证据内容，即答案中的引用是否指向真实检索到的文档。

**计算公式**：
```
faithfulness = 有效引用数 / 总引用数

有效引用 = 答案中 [doc_xxx] 格式的引用，且 xxx 存在于检索结果的 doc_id 列表中
```

**计算逻辑**：
1. 用正则表达式 `\[doc_\w+\]` 从答案中提取所有引用标记
2. 将每个引用标记的 doc_id 与检索结果的 doc_id 列表比对
3. 如果没有检索到证据但答案诚实地说"暂无匹配"，给予 0.9 分
4. 如果检索到证据但答案没有任何引用标记，给予 0.3 分（有证据但未引用）

**评分边界**：

| 场景 | 评分 |
|------|------|
| 无检索证据 + 答案含"暂无匹配"/"无法给出" | 0.9 |
| 有检索证据 + 引用全部有效 | 1.0 |
| 有检索证据 + 部分引用无效 | 0.0~1.0 |
| 有检索证据 + 无任何引用 | 0.3 |
| 无检索证据 + 答案编造内容 | 0.0 |

**达标阈值**：≥ 0.8（80%）

### 3.3 指标2：引用准确率（Citation Accuracy）

**定义**：衡量答案中引用标记的格式规范性，即引用是否遵循 `[doc_xxx]` 的标准格式。

**计算公式**：
```
citation_accuracy = 格式正确的引用数 / 总引用数

格式正确 = 引用以 [doc_ 开头、以 ] 结尾
```

**计算逻辑**：
1. 用正则表达式提取答案中的所有引用标记
2. 检查每个引用是否符合 `[doc_` 开头 + `]` 结尾的格式
3. 如果没有检索到证据，给予 0.9 分（不引用比错误引用好）
4. 如果检索到证据但无引用，给予 0.0 分

**与忠实度的区别**：
- **忠实度**关注引用的**内容正确性**（指向的文档是否真实存在）
- **引用准确率**关注引用的**格式规范性**（格式是否符合规范）

**达标阈值**：≥ 0.85（85%）

### 3.4 指标3：检索精准率（Context Precision）

**定义**：衡量检索系统返回的结果与查询的相关程度。

**计算方式**（分两种情况）：

**情况A：测试集提供了 `expected_chunks`（预期文档列表）**
```
context_precision = 命中的预期文档数 / 预期文档总数

命中 = 预期文档ID出现在检索结果的doc_id列表中
```

**情况B：测试集未提供 `expected_chunks`（当前默认情况）**
```
context_precision = min(平均检索相似度分数, 1.0)

平均检索相似度 = sum(chunk.score for chunk in chunks) / len(chunks)
```

**计算逻辑**：
- 当有标准答案对应的预期文档时，直接计算命中率
- 当无预期文档时，使用检索结果的相似度分数（BM25+向量混合检索的加权融合分数）作为代理指标
- 分数被限制在 [0, 1] 范围内

**达标阈值**：≥ 0.75（75%）

### 3.5 指标4：答案相关性（Answer Relevance）

**定义**：衡量 LLM 生成的答案与标准答案（ground truth）的内容重合度。

**计算公式**：
```
answer_relevance = |答案关键词 ∩ 标准答案关键词| / |标准答案关键词|

关键词提取 = 正则匹配中文连续字符段 + 英文单词
```

**计算逻辑**：
1. 用正则 `[\u4e00-\u9fa5]+|[a-zA-Z]+` 分别从答案和标准答案中提取关键词
2. 计算两个关键词集合的交集大小
3. 交集大小除以标准答案关键词集合大小

**特点**：
- 基于关键词重合度，计算简单高效
- 不依赖外部模型（如 BERTScore），适合快速评测
- 局限性：无法捕捉语义等价但用词不同的答案（如"首选"vs"一线推荐"）

**达标阈值**：≥ 0.8（80%）

### 3.6 达标率（Pass Rate）

**定义**：同时满足证据忠实度和引用准确率两项阈值的题目占比。

**计算公式**：
```
pass_rate = count(faithfulness >= 0.8 AND citation_accuracy >= 0.85) / total_count
```

**设计理由**：达标率只考察生成质量的两项硬性指标（忠实度+引用准确率），不纳入检索精准率和答案相关性，因为后两者更多取决于测试集本身的质量和难度，而非系统稳定性。

### 3.7 指标阈值配置

所有阈值通过环境变量可配置，默认值定义在 `backend/config.py`：

| 指标 | 环境变量 | 默认值 | 说明 |
|------|---------|--------|------|
| 证据忠实度 | `FAITHFULNESS_MIN` | 0.80 | 答案中80%以上的引用须指向真实文档 |
| 引用准确率 | `CITATION_ACC_MIN` | 0.85 | 答案中85%以上的引用须格式正确 |
| 检索精准率 | `CONTEXT_PRECISION_MIN` | 0.75 | 检索结果平均相似度须≥0.75 |
| 答案相关性 | `ANSWER_RELEVANCE_MIN` | 0.80 | 答案须覆盖80%以上的标准答案关键词 |

## 四、评测原理

### 4.1 RAG 评测的理论基础

RAG 系统的评测与传统 LLM 评测有本质区别。传统 LLM 评测（如 MMLU、MedQA）关注模型的"闭卷"知识能力，而 RAG 评测需要同时评估**检索**和**生成**两个环节。

本系统的评测框架参考了 RAGAS（Retrieval Augmented Generation Assessment）框架的核心思想，将评测维度分解为：

```
用户提问 → [检索环节] → 检索到的证据 → [生成环节] → 最终答案
              ↓                              ↓
        检索精准率                    证据忠实度
                                      引用准确率
                                      答案相关性
```

### 4.2 检索环节评测

检索环节使用**混合检索**策略：BM25（关键词匹配）+ 向量检索（语义相似度），加权融合后通过 CrossEncoder 重排序。

**检索精准率**的评估原理：
- 当有 `expected_chunks` 时，直接评估检索系统是否找回了"正确"的文档（召回率视角）
- 当无 `expected_chunks` 时，使用检索结果的相似度分数作为代理指标（置信度视角），分数越高表示检索结果与查询越相关

### 4.3 生成环节评测

生成环节评估 LLM 是否基于检索到的证据生成答案，而非凭空编造（幻觉）。

**证据忠实度**的评估原理：
- 核心思想：答案中的每个引用都应该指向一个真实检索到的文档
- 如果答案引用了不存在的文档，说明 LLM 可能产生了幻觉
- 如果答案完全没有引用，说明 LLM 未有效利用检索结果

**引用准确率**的评估原理：
- 核心思想：引用格式必须规范，便于读者溯源验证
- 标准格式：`[doc_xxx]`，其中 xxx 是文档ID
- 格式不规范的引用会导致读者无法定位证据来源

### 4.4 为什么需要两类测试集

| 维度 | 域内测试集 | 域外测试集 |
|------|-----------|-----------|
| **答案来源** | 知识库文献本身 | 权威临床指南 |
| **评估重点** | 检索召回率、引用规范性 | 临床推理能力、知识覆盖度 |
| **优势** | 可自动生成、可大规模扩展 | 评估更全面、更贴近真实临床场景 |
| **局限** | 无法评估系统的医学推理能力 | 题量有限、构建成本高 |
| **适用场景** | 持续集成测试、回归测试 | 系统级评估、性能基线建立 |

两类测试集互补使用：域内测试集验证"系统能否找到知识库中的内容"，域外测试集验证"系统能否正确回答临床问题"。

### 4.5 评测器架构

系统实现了两级评测器：

| 评测器 | 依赖 | 适用场景 |
|--------|------|---------|
| `SimpleEvaluator` | 无外部依赖 | 默认使用，基于规则的关键词匹配和正则提取 |
| `RagasEvaluator` | ragas库 | 可选安装，支持基于LLM的语义级评测 |

当前默认使用 `SimpleEvaluator`，其优势是计算快速、无需额外API调用。`RagasEvaluator` 作为可选升级，支持更深层次的语义评估（如基于LLM的答案一致性判断），但需要额外的LLM API调用开销。

| 题目 | 评分 | 成功原因 |
|------|------|---------|

### 5.3 失败题目分析

| 题目 | 评分 | 失败原因 |
|------|------|---------|
| 冠心病ABCDE策略 | 0.330 | 答案未完整列出A-E五个字母对应的措施，关键信息覆盖不足 |
| 代谢综合征诊断标准 | 0.434 | 缺少腰围、血压、血糖、TG、HDL-C五项具体数值 |
| 高血压急症降压原则 | 0.425 | 未覆盖"1小时内降25%、2-6小时降至160/100"的关键时序 |

**失败原因归类**：
- **答案细节覆盖不足**（2/3题）：系统能理解问题方向，但缺少精确的数值和分层标准
- **结构化输出不足**（1/3题）：对于需要逐项列举的问题（如ABCDE策略），答案组织不够结构化

### 5.4 指标解读

**证据忠实度 0.689（未达标）**：部分答案的引用标记不够规范，或引用了检索结果中不存在的文档ID。改进方向：优化 LLM 的系统提示词，强调必须使用 `[doc_xxx]` 格式引用。

**引用准确率 0.667（未达标）**：与忠实度问题相关，部分引用格式不规范。改进方向：在答案生成后增加引用格式校验和自动修复步骤。

**检索精准率 0.740（接近达标）**：混合检索+重排的总体效果良好，但部分题目的检索相似度分数偏低。改进方向：调优 BM25/向量权重比例，或增加检索结果数量。

**答案相关性 0.700（接近达标）**：基于关键词重合度的评估显示答案覆盖了大部分标准答案内容，但对于用词不同但语义相同的答案可能低估。改进方向：引入语义相似度评估（如 BERTScore）。

## 六、评测系统使用指南

### 6.1 前端操作流程

1. **准备测试集**：
   - 上传：点击"上传测试集"按钮，上传 JSON 格式的测试集文件
   - 生成：点击"生成测试集样例"按钮，设置题数，系统自动基于知识库生成

2. **运行评测**：
   - 在测试集列表点击"运行评测"，或在评测配置区选择测试集+设置条目数后点击"开始评测"
   - 评测完成后，结果自动展示在"评测报告"区域，同时保存到数据库

3. **查看历史**：
   - "评测历史记录"表格展示所有历史评测，点击任意行或"详情"按钮查看完整结果
   - 历史详情弹窗展示概览指标+每题的详细结果（问题、标准答案、系统回答、各项评分）

4. **导出报告**：
   - 在当前评测结果区域点击"导出报告"，将评测报告保存为 JSON 文件

### 6.2 后端API

| 方法 | 路径 | 功能 |
|------|------|------|

### 6.3 数据持久化

评测结果存储在 SQLite 数据库 `table_eval_history` 表中：

| 字段 | 类型 | 说明 |
|------|------|------|
| eval_id | TEXT | 评测记录唯一ID |
| test_set_filename | TEXT | 使用的测试集文件名 |
| total_count | INTEGER | 测试题目总数 |
| avg_faithfulness | REAL | 平均证据忠实度 |
| avg_citation_accuracy | REAL | 平均引用准确率 |
| avg_context_precision | REAL | 平均检索精准率 |
| avg_answer_relevance | REAL | 平均答案相关性 |
| pass_rate | REAL | 达标率 |
| details | TEXT | 完整评测结果（JSON格式，含每题详情） |
| timestamp | TEXT | 评测时间 |
